# Задание 3. Проектирование целевой архитектуры и оценка рисков внедрения

1. Спроектируйте целевую архитектуру системы «Будущего 2.0» в горизонте трёх лет, используя C4-модель на уровне контейнеров и компонентов. Покажите, как изменятся ключевые системы компании при масштабировании, включая появление новых бизнес-направлений, интеграцию дополнительных источников данных и отказ от легаси-систем.

2. Составьте карту рисков трансформации, определив архитектурные, организационные и технологические риски. Для каждого риска укажите вероятность его возникновения (низкая, средняя, высокая) и потенциальное влияние на бизнес (низкое, среднее, высокое).

3. Предложите план управления рисками, описав конкретные меры по их снижению. Укажите, какие риски можно минимизировать с помощью технических решений, а какие — управленческими подходами.

# Решение

## 1. Целевая архитектура

Через 3 года система должна прийти к:
- Event-driven architecture (EDA)
- Data Mesh (доменные витрины данных)
- Микросервисная архитектура и стримы данных для аналитики
- Cloud-native (Kubernetes + managed services)
- На контейнеры согласно схемы C4 (см. ниже)
- На основные домены (см. ниже)

### Описание архитектуры

#### 1. Основные домены

- Medical (пациенты, лечение, диагностика)
- Fintech (счета, кредиты, платежи)
- AI (диагностика, модели)
- Inventory & Devices (оборудование)
- HR
- Analytics / Data Platform~

#### 2. API слой

Ключевые контейнеры
- API слой
- API Gateway (Kong, Istio...)
- BFF (Backend for Frontend)

#### 3. Доменные микросервисы

Каждый домен должен иметь:
- собственную БД
- самостоятельно публикавать свои события
- иметь свой bounded context

####  4. Event Streaming Platform включает:
Внедрить:
- Kafka
- DLQ (Dead Letter Queue)
- Retention + replay

Использовать гибридный подход для ETL:
1. Batch ETL (остается)
    - Airflow (используется для):
        - исторических данных
        - регуляторной отчётности
        - сложных пересчётов
2. Streaming ETL (новое)
    - Flink / Kafka Streams (используется для):
    - real-time витрин
    - событийной аналитики

#### 5. Data Platform (Data Mesh)

Каждый домен владеет:
- Data Product
- Потоковые витрины (streaming marts)
- Слои:
    Ingestion — события из Kafka
    Processing — Flink / Spark Streaming
- Храниние:
    Data Lake (S3 / ADLS)
    OLAP (ClickHouse / BigQuery / Snowflake)
- Обслуживание: API + BI

####  6. BI и Self-Service
- Power BI (или замена)
- Semantic Layer
- Data Catalog

####  7. AI Platform
- Feature Store
- Model Registry
- Inference services (real-time + batch)

#### 8. Legacy Bridge
- Anti-Corruption Layer
- CDC (Change Data Capture)
- Постепенное "высасывание" данных из DWH

#### 9. Как изменится текущий ландшафт

| Сейчас | Через 3 года |
|--------|--------------|
| DWH (центр всего) | Data Mesh (децентрализация) |
| Batch-обработка | real-time-обработка |
| Camel ESB | Event Streaming |
| PowerBuilder | Web / API-first |

#### 10. Деклмпозиция ESB

| Роутинг | Kafka (topics)| 
|--------|--------------|
| Транспорт | Kafka / API Gateway| 
| Синхронные интеграции | API Gateway| 
| Асинхронные интеграции | Event Streaming| 
| ФЛК (валидация) | Schema Registry + доменные сервисы| 
| пакетные рансформации (batch ETL)| ETL-пайплайны (Airflow)| 
| Стриминговые трансформации | Flink / Kafka Streams| 
| Оркестрация | Saga / хореография| 
| Бизнес-логика | доменные микросервисы| 
| Интеграция с legacy | Anti-Corruption Layer|

## 3. План управления рисками

| Риск | Меры по снижению  |
|------|-------------------|
|Сложность миграции DWH	| - StranglerFig (постепенная миграция)<br/>- CDC (Debezium) |
|Дублирование данных (Data Mesh)| - версионирование событий, аггрегаты<br/>- внедрение Schema registry |
|Нарушение консистентности|	- реализация SAGA-паттерна для интеграций<br/>- поддержка иденпотентности<br/>- версионирование событий |
|Перегрузка Kafka / streaming| - DLQ<br/>- Retry policy<br/>- потоковая обработка<br/>- разделение данных по важности |
|Недостаток экспертизы в EDA, Сопротивление команд| - обучение<br/>- расширение штата<br/>- реализация автодокументирования по Data mesh|
|Проблемы с безопасностью (PII/мед данные)| - Шифрование, маскирование, токенизация данных<br/>- Пересмотр и настройка ролевой модели |
|Несоответствие регуляторам|- минимизация, маскирование, токенизация данных<br/>- внедрение роли "Data Product Owner"<br/>- Создание архитектурного комитета по данным|
|Рост стоимости облака| - регулярный мониторинг расходов, техническая оптимизация|
|Сложность мониторинга распределённой системы| - внедрение Jaeger-трейсинга<br/>- prometheus-метрики от всех систем<br/>- централизованное хранилище логов (ELK/Splunk) |
|- Деградация качества данных<br/>- Недостаточный контроль схем (без ESB) | - ведение документации по контрактам данных<br/>- Schema Registry<br/>- пайплайны валидации данных |
|Vendor lock-in| - внедрение подходов VendorAgnostic|
| - Потеря логики при декомпозиции ESB<br/>- Скрытая бизнес-логика в Camel<br/>- Дублирование трансформаций| - инвентаризация всех маршрутов ESB (классификация, маршруты, трансформации, бизнес-логика)|
| Неконсистентные данные при параллельной работе| - тестирование (проверка контрактов, replay событий)| 
| Разрыв интеграций при отключении ESB| - двойная запись + сравнение результатов (в старую и новую систему, временно)|
