# Задание 3. Проектирование целевой архитектуры и оценка рисков внедрения

1. Спроектируйте целевую архитектуру системы «Будущего 2.0» в горизонте трёх лет, используя C4-модель на уровне контейнеров и компонентов. Покажите, как изменятся ключевые системы компании при масштабировании, включая появление новых бизнес-направлений, интеграцию дополнительных источников данных и отказ от легаси-систем.

2. Составьте карту рисков трансформации, определив архитектурные, организационные и технологические риски. Для каждого риска укажите вероятность его возникновения (низкая, средняя, высокая) и потенциальное влияние на бизнес (низкое, среднее, высокое).

3. Предложите план управления рисками, описав конкретные меры по их снижению. Укажите, какие риски можно минимизировать с помощью технических решений, а какие — управленческими подходами.

# Решение

## 1. Целевая архитектура

Через 3 года система должна прийти к:
- Event-driven architecture (EDA)
- Data Mesh (доменные витрины данных)
- Микросервисная архитектура и стримы данных для аналитики
- Cloud-native (Kubernetes + managed services)

## 3. План управления рисками

| Риск | Меры по снижению  |
|------|-------------------|
|Сложность миграции DWH	| - StranglerFig (постепенная миграция)<br/>- CDC (Debezium) |
|Дублирование данных (Data Mesh)| - версионирование событий, аггрегаты<br/>- внедрение Schema registry |
|Нарушение консистентности|	- пайплайны валидации данных<br/> - реализация SAGA-паттерна для интеграций |
|Перегрузка Kafka / streaming| - потоковая обработка<br/>- разделение данных по важности |
|Недостаток экспертизы в EDA| - обучение<br/>- расширение штата<br/>- реализация автодокументирования по Data mesh|
|Проблемы с безопасностью (PII/мед данные)| - Шифрование, маскирование, токенизация данных<br/>- Пересмотр и настройка ролевой модели |
|Несоответствие регуляторам|- минимизация, маскирование, токенизация данных|
|Рост стоимости облака| - регулярный мониторинг расходов, техническая оптимизация|
|Сложность мониторинга распределённой системы| - внедрение Jaeger-трейсинга<br/>- prometheus-метрики от всех систем<br/>- централизованное хранилище логов (ELK/Splunk) |
|Data quality деградация| - внедрение роли "Data Product Owner"<br/>- Создание архитектурного комитета по данным|
|Vendor lock-in| - внедрение подходов VendorAgnostic|
